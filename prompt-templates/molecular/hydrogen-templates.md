# 氢键模板（Hydrogen Bond Templates）

> **目标**：人工诱导反思机制，主动触发验证和修正
> 
> **适用场景**：需要高准确性的问题、容易出错的任务、需要验证的推理

---

## 模板 H1: 负反馈回路（Negative Feedback Loop）

### 设计原理

在每个关键步骤后主动寻找错误，通过负面思考提高准确性。

### 模板内容

```
【目标】在每个关键步骤后主动寻找错误

【指令】
采用"怀疑一切"的思维方式：

每完成一个推理步骤后，立即进行"反审计"：
1. 【假设反转】如果这个结论是错的，可能因为什么？
2. 【边界测试】这个结论在什么情况下不成立？
3. 【量级检查】结果的数量级是否合理？

【格式】
推理步骤: [内容]
├─ 反审计 1: [假设反转] ...
├─ 反审计 2: [边界测试] ...
└─ 反审计 3: [量级检查] ...
   └─ 结论: [继续/修正/回退]

只有当三个反审计都通过时，才能进入下一步。

【示例】
问题：一个正方形的边长是5cm，求面积。

推理步骤: 面积 = 5 × 5 = 25 cm²
├─ 反审计 1: [假设反转] 如果边长不是5，会怎样？
│   └─ 答：那面积就不是25，但题目明确是5
├─ 反审计 2: [边界测试] 极端情况下公式还成立吗？
│   └─ 答：边长接近0时面积接近0，符合直觉
└─ 反审计 3: [量级检查] 25cm²合理吗？
   └─ 答：一个5cm边长的正方形，面积约一张小卡片，合理
   └─ 结论: 继续确认答案为25cm²

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：计算问题、需要高准确性的任务
- **预期行为分布**：Reflection ~40%, Deep Reasoning ~40%
- **注意事项**：会显著增加推理时间

---

## 模板 H2: 反向质疑（Reverse Questioning）

### 设计原理

从结论倒推验证，类似数学中的逆向证明。

### 模板内容

```
【目标】从结论倒推，验证推理的可靠性

【指令】
在每个阶段性结论后，执行"反向提问"：

1. 逆向追溯：这个结论需要哪些前提条件？
2. 前提验证：这些前提条件是否都成立？
3. 唯一性检查：是否有其他路径也能得到这个结论？

【格式】
结论: [X]
└─ 逆向提问
   ├─ Q1: 得到X需要什么条件？
   │   └─ A1: [条件A, 条件B, ...]
   ├─ Q2: 这些条件都成立吗？
   │   └─ A2: [验证过程]
   └─ Q3: 还有其他方式得到X吗？
       └─ A3: [探索其他路径]

如果发现任何不一致，回到相应步骤重新推理。

【示例】
问题：证明 √2 是无理数。

中间结论: 假设 √2 = p/q（p,q互质）会导致矛盾
└─ 逆向提问
   ├─ Q1: 得到这个结论需要什么条件？
   │   └─ A1: 需要有理数的定义、互质的概念
   ├─ Q2: 这些条件都成立吗？
   │   └─ A2: 是的，这是有理数的标准定义
   └─ Q3: 还有其他方式得到矛盾吗？
       └─ A3: 可以从 p²=2q² 推出 p 是偶数，进而推出 q 也是偶数，
             与互质矛盾

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：证明题、逻辑推理、需要验证的结论
- **预期行为分布**：Reflection ~35%, Deep Reasoning ~45%
- **注意事项**：对简单问题可能过度复杂化

## 模板 H3: 漏洞扫描（Vulnerability Scanner）

### 设计原理

将推理过程视为程序，进行系统性的"代码审查"。

### 模板内容

```
【目标】系统性地扫描推理链条中的薄弱环节

【指令】
将推理过程视为一个程序，执行"代码审查"：

在完成推理后，进行以下扫描：

【扫描清单】
□ 未定义的变量（模糊的概念/术语）
□ 循环依赖（A依赖B，B又依赖A）
□ 越界访问（引用不存在的前提）
□ 类型不匹配（概念混用）
□ 空指针（无依据的断言）
□ 精度丢失（过度近似）
□ 逻辑跳跃（步骤间缺乏连接）

【格式】
推理完成后的审计报告：
├─ [扫描项1]: ✅ 通过 / ❌ 发现问题
│   └─ 详情: ...
├─ [扫描项2]: ...
└─ 总体评估: [高风险/中风险/低风险]

如果发现高风险问题，必须修正后再给出最终答案。

【示例】
问题：一个人从A地到B地需要2小时，返回需要3小时，求平均速度。

推理：平均速度 = (速度1 + 速度2) / 2 = (d/2 + d/3) / 2

审计报告：
├─ 未定义变量: ✅ 通过 (d 表示距离)
├─ 循环依赖: ✅ 通过 (无循环)
├─ 越界访问: ✅ 通过 (所有变量已定义)
├─ 类型不匹配: ✅ 通过 (都是速度单位)
├─ 空指针: ✅ 通过 (每步有依据)
├─ 精度丢失: ✅ 通过 (使用分数)
├─ 逻辑跳跃: ❌ 发现问题
│   └─ 详情: 平均速度的公式错误！
│            应该是 总路程/总时间 = 2d/(2+3) = 2d/5
│            而不是速度的平均
└─ 总体评估: 高风险 - 需要修正

修正后答案：平均速度 = 2d / 5 = 2 × (d/2) / 5 × ... = 需要知道d

实际上如果只给时间不给距离，答案应该用调和平均：
v_avg = 2 × v1 × v2 / (v1 + v2)，但这里用距离更直接。

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：所有需要严谨推理的问题
- **预期行为分布**：Reflection ~30%, Deep Reasoning ~50%
- **注意事项**：可能发现"假阳性"问题

## 模板 H4: 信心校准（Confidence Calibration）

### 设计原理

要求模型为每个步骤和结论给出置信度，促进自我评估。

### 模板内容

```
【目标】通过置信度标注，促进模型的自我评估

【指令】
在推理过程中，为每个关键陈述标注置信度：

【置信度定义】
- [100%] 数学确定性/已知事实
- [80-99%] 高度确定的逻辑推论
- [60-79%] 合理但有验证空间的推论
- [40-59%] 基于直觉或部分信息的猜测
- [20-39%] 不确定，需要更多信息
- [0-19%] 纯粹猜测

【格式】
[n] [置信度] 内容
    └─ 依据: [为什么是这个置信度]
    └─ 风险: [什么情况下会错]

【最终答案要求】
- 只有当最终答案置信度 ≥ 80% 时，才给出确定答案
- 置信度 60-79% 时，给出"暂定答案"并说明不确定性
- 置信度 < 60% 时，说明需要更多信息

【示例】
问题：太阳系有几颗行星？

[1] [100%] 冥王星在2006年被重新分类为矮行星
    └─ 依据: 这是IAU的官方决定，广泛报道
    └─ 风险: 几乎无风险
    
[2] [100%] 太阳系的行星是：水星、金星、地球、火星、木星、土星、天王星、海王星
    └─ 依据: 基于天文学定义
    └─ 风险: 几乎无风险

最终答案: [95%] 太阳系有8颗行星

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：知识性问题、需要可靠性评估的任务
- **预期行为分布**：Reflection ~35%, Deep Reasoning ~45%
- **注意事项**：模型可能高估自己的置信度
