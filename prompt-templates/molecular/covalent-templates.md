# 共价键模板（Covalent Bond Templates）

> **目标**：强化逻辑主干的强制性，构建严密的推理链条
> 
> **适用场景**：数学推理、逻辑证明、需要严密逻辑的问题

---

## 模板 C1: 公理化推理（Axiomatic Reasoning）

### 设计原理

模仿数学证明的公理化方法，要求每个推理步骤都有明确依据。

### 模板内容

```
【目标】构建严密的逻辑链条，每一步都有明确的逻辑依据

【指令】
请按以下公理化方法进行推理：

1. **定义公理层**
   - 列出已知事实和前提（不需要证明）
   - 标记为 [AXIOM]

2. **构建定理层**
   - 每个推理步骤必须引用前面的事实
   - 使用格式：[STEP-N] 因为 [引用], 所以 [结论]
   
3. **保持因果链完整性**
   - 如果某一步骤无法找到依据，标记 [GAP] 并说明
   - 不允许跳跃式推理

【示例】
问题：小明比小红大3岁，小红今年10岁，小明去年多大？
[AXIOM-1] 小红今年10岁
[AXIOM-2] 小明比小红大3岁
[STEP-1] 因为 [AXIOM-1][AXIOM-2]，所以 小明今年 = 10 + 3 = 13岁
[STEP-2] 因为 [STEP-1]，所以 小明去年 = 13 - 1 = 12岁

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：数学计算、逻辑推理、因果分析
- **预期行为分布**：Deep Reasoning ~70%
- **注意事项**：对复杂问题可能导致步骤过多

## 模板 C2: 因果链标注（Causal Chain Annotation）

### 设计原理

明确标注因果关系，使推理过程可追溯。

### 模板内容

```
【目标】明确标注每一步的因果关系，形成可追溯的推理链

【指令】
在推理过程中，请为每个陈述标注其因果类型：

→ [CAUSE] 原因陈述
→ [EFFECT] 由上一条原因导致的结果
→ [BIDIR] 互为因果
→ [INDEPENDENT] 独立陈述（无因果关联）

每个 [EFFECT] 必须明确指向其 [CAUSE]

【格式要求】
步骤N: [类型] 内容 [← 引用步骤M]

【示例】
问题：为什么下雨天路面会湿？
步骤1: [CAUSE] 下雨时水滴从天空落下 [独立事实]
步骤2: [EFFECT] 水滴落在路面上 [← 步骤1]
步骤3: [EFFECT] 路面被水覆盖 [← 步骤2]
步骤4: [EFFECT] 光线在水面上反射，看起来湿润 [← 步骤3]

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：因果分析、物理现象解释、事件推演
- **预期行为分布**：Deep Reasoning ~65%
- **注意事项**：不适合复杂的多因果问题

---

## 模板 C3: 步骤引用强制（Enforced Step Reference）

### 设计原理

强制模型引用前序步骤，防止幻觉和跳跃推理。

### 模板内容

```
【目标】强制模型引用之前的步骤，防止幻觉

【指令】
解决此问题时，每个新步骤必须：
1. 明确引用之前的1-3个步骤
2. 说明引用的原因
3. 如果无法引用，说明这是"新起点"

【格式】
[n] 内容
    └─ 基于: [引用步骤] 
    └─ 关系: [如何相关]
    └─ 推导: [具体逻辑]

【禁止】
- 不允许没有引用的跳跃结论
- 不允许引用不存在或逻辑无关的步骤

【示例】
问题：一个数的平方等于它本身，这个数是多少？
[1] 这个问题需要找到 x 使得 x² = x
    └─ 基于: [问题陈述]
    └─ 关系: 直接转述
    └─ 推导: 数学等式
    
[2] 移项得 x² - x = 0
    └─ 基于: [1]
    └─ 关系: 等式变换
    └─ 推导: 两边减去x
    
[3] 因式分解得 x(x-1) = 0
    └─ 基于: [2]
    └─ 关系: 代数恒等
    └─ 推导: 提取公因子x

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：所有需要严密推理的问题
- **预期行为分布**：Deep Reasoning ~75%
- **注意事项**：输出较长，适合重要问题

---

## 模板 C4: 层次化推理（Hierarchical Reasoning）

### 设计原理

将复杂问题分解为层次结构，逐层解决。

### 模板内容

```
【目标】将复杂问题分解为可管理的子问题层次

【指令】
采用层次化方法解决问题：

【Level 0: 问题定义】
- 明确要解决的核心问题
- 定义成功标准

【Level 1: 子问题分解】
- 将核心问题分解为2-5个子问题
- 每个子问题应该独立可解
- 标记子问题间的依赖关系

【Level 2: 子问题求解】
- 按依赖顺序解决每个子问题
- 记录每个子问题的解

【Level 3: 整合验证】
- 将子问题的解整合为最终答案
- 验证答案满足Level 0的成功标准

【格式】
## [L0] 核心问题
...

## [L1] 子问题分解
├─ 子问题1: ... (依赖: 无)
├─ 子问题2: ... (依赖: 子问题1)
└─ 子问题3: ... (依赖: 子问题1)

## [L2] 子问题求解
### 子问题1
[L2-1] ...

### 子问题2
[L2-2] ...

## [L3] 整合验证
...

现在请解决以下问题：
{question}
```

### 使用说明

- **适用问题类型**：复杂多步骤问题、系统设计问题
- **预期行为分布**：Deep Reasoning ~60%, Exploration ~20%
- **注意事项**：需要问题本身具有层次结构
