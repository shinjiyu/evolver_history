# Peak Hours Monitoring - 高峰期监控策略

在 API 高峰期实施预测性监控和预防措施，确保系统稳定性。

## 核心理念

**预防 > 监控 > 响应**

在问题发生前采取预防措施，而非被动响应。

## 监控策略

### 阶段 1: 准备期 (06:00-07:00)

**目标**: 在高峰期前识别潜在问题

**检查项**:
- [ ] 系统资源使用率 (CPU/内存/磁盘)
- [ ] 活跃任务数量
- [ ] 最近 1 小时错误率
- [ ] 外部 API 服务状态

**行动阈值**:
```
如果错误率 > 3/h:
  → 降低非关键任务频率 50%
  → 准备应急响应方案
```

### 阶段 2: 高峰期 (07:00-09:00)

**目标**: 实时监控，快速响应

**监控频率**:
- 5 分钟检查：最近 5 分钟错误
- 15 分钟检查：最近 15 分钟错误
- 30 分钟检查：最近 30 分钟错误

**警戒阈值**:

| 指标 | 正常 | 警戒 | 危险 | 行动 |
|------|------|------|------|------|
| 429/5m | 0 | 1 | > 2 | 立即响应 |
| 429/15m | 0 | 1-3 | > 3 | 调整频率 |
| 429/30m | 0 | 1-5 | > 5 | 暂停任务 |

**应急响应**:

```
Level 1 (1-3 次/30m):
  → 继续监控
  → 准备降级方案

Level 2 (3-5 次/30m):
  → 暂停非关键任务 (evover-log-analysis, evomap-outreach)
  → 降低高频任务频率 50%

Level 3 (> 5 次/30m):
  → 暂停所有非关键任务
  → 仅保留 adaptive-scheduler (最低频率)
  → 等待 15 分钟后评估
```

### 阶段 3: 评估期 (09:00-10:00)

**目标**: 总结经验，优化策略

**评估项**:
- [ ] 高峰期总错误数
- [ ] 与历史对比
- [ ] 成功/失败因素分析
- [ ] 改进建议

## 健康评分模型

### 评分公式

```
健康评分 = 基础分 + 奖励分 - 惩罚分

基础分: 8.0/10
奖励分:
  + 高峰期无错误: +1.0
  + 关键时段无错误: +0.5
惩罚分:
  - 历史错误率 (6h): -0.5 (15.2/h)
  - 当前错误: -1.0 (每次)
```

### 评分趋势示例

```
06:00 → 7.0/10 (高峰期预警)
06:30 → 6.5/10 (高峰期临近)
07:00 → 7.5/10 (高峰期开始，无错误)
07:30 → 9.0/10 (高峰期过半，持续无错误)
08:00 → 9.0/10 (关键时段，0 错误)
09:00 → 9.0/10 (高峰期结束，评估)
```
