# Evolution Verification - 进化效果验证

验证自进化措施的效果，建立「发现问题 → 修复 → 验证 → 再改进」的完整闭环。

## 核心流程

```
读取优化记录 → 收集对比数据 → 分析效果 → 生成验证报告 → 建议下一步
```

## 使用场景

### 1. 执行优化后验证

```
用户：验证 Round 168 的优化效果
```

执行：
1. 读取 Round 168 的进化报告
2. 收集优化前后的指标
3. 对比预期效果 vs 实际效果
4. 生成验证报告

### 2. 定期健康检查

```
用户：检查系统健康度
```

执行：
1. 分析最近的错误趋势
2. 检查关键指标（API 错误率、任务成功率等）
3. 对比历史基线
4. 生成健康评分

### 3. 问题趋势分析

```
用户：分析错误趋势
```

执行：
1. 统计不同类型错误的频率
2. 识别恶化/改善的模式
3. 预测潜在风险
4. 建议预防措施

## 验证指标

### 短期指标（0-6 小时）

| 指标 | 说明 | 目标 |
|------|------|------|
| API 错误率 | 4xx/5xx 错误占比 | < 5% |
| Rate Limit 频率 | 每小时 429 错误数 | < 2/hour |
| 任务成功率 | 成功任务占比 | > 95% |
| Edit 失败率 | Edit 工具失败占比 | < 5% |
| ENOENT 错误 | 文件不存在错误数 | < 2 次 |

### 中期指标（1-3 天）

| 指标 | 说明 | 目标 |
|------|------|------|
| 日志噪音 | 无价值的分析报告数 | < 10% |
| 系统稳定性 | 连续无错误的小时数 | > 6 hours |
| 资源使用 | CPU/内存平均使用率 | < 70% |
| 分析价值密度 | 发现新问题的报告占比 | > 30% |

### 长期指标（1-2 周）

| 指标 | 说明 | 目标 |
|------|------|------|
| 模式重复率 | 重复出现的问题模式 | 下降趋势 |
| Skill 可用性 | 新 Skill 被使用次数 | > 5 次 |
| 自动修复率 | 无需人工干预的修复 | > 80% |
| 系统健康评分 | 综合健康度 | > 9.0/10 |

```bash
# - 优化前：Round 168 之前的数据

```

### Step 3: 分析效果

**计算改进幅度**：
```
改进幅度 = ((优化前 - 优化后) / 优化前) * 100%

示例：
- 429 错误：优化前 206 次/6小时，优化后 18 次/2小时
- 标准化：206/6 = 34.3/hour, 18/2 = 9/hour
- 改进幅度 = ((34.3 - 9) / 34.3) * 100% = 73.7%
```

**评估效果等级**：
- 🟢 **显著改善**: 改进 > 70%
- 🟡 **明显改善**: 改进 30-70%
- 🟠 **轻微改善**: 改进 10-30%
- 🔴 **无效果**: 改进 < 10% 或恶化
