# API Balance Monitor - API 余额监控策略

实时监控 API 余额，在余额不足前采取预防措施，避免服务中断。

## 核心理念

**预警 > 监控 > 降级 > 恢复**

在余额耗尽前预警，在余额不足时自动降级，避免全面中断。

## 问题背景

### PAT-025：API 余额耗尽危机

**发现时间**: 2026-02-26 04:00

| 指标 | 数值 | 说明 |
|------|------|------|
| 错误次数 | **136+** | 余额不足错误 |
| 影响时段 | 03:40-04:50 | 高频触发 |
| 系统健康评分 | 9.5 → **6.5** | 降至危险等级 |
| 影响 | **所有任务失败** | 全面中断 |

**错误信息**:
```
429 余额不足或无可用资源包,请充值。
429 您的账户已达到速率限制，请您控制请求频率
```

**根因分析**:
1. **余额耗尽** - 智谱 AI GLM-5 账户余额不足
2. **缺乏预警** - 没有余额监控机制
3. **自动降级缺失** - 余额低时未自动减少任务
4. **通知缺失** - 余额耗尽时未及时通知

## 监控策略

### 阶段 1: 预警期（余额 > 20%）

**目标**: 提前发现余额不足风险

**检查项**:
- [ ] API 余额百分比
- [ ] 最近 24 小时消耗速度
- [ ] 预计耗尽时间
- [ ] 历史消耗模式

**行动阈值**:
```
如果余额 < 30%:
  → 发送预警通知
  → 记录到 memory/api-balance-alert.md
  
如果余额 < 20%:
  → 降低非关键任务频率 50%
  → 启用成本优化模式
  → 每 4 小时检查一次
```

### 阶段 2: 警戒期（余额 10-20%）

**目标**: 延缓余额耗尽，准备应急方案

**行动阈值**:
```
如果余额 < 20%:
  → 暂停非关键任务
  → 仅保留核心任务（adaptive-scheduler, evolver-self-evolution）
  → 每 2 小时检查一次
  → 发送紧急通知

如果余额 < 15%:
  → 降低所有任务频率 75%
  → 准备充值
```

### 阶段 3: 危急期（余额 < 10%）

**目标**: 避免全面中断，最小化影响

**应急响应**:
```
如果余额 < 10%:
  → 暂停所有任务（除 adaptive-scheduler 最低频率）
  → 发送紧急通知
  → 等待充值
  → 每 1 小时检查一次

如果余额 = 0:
  → 系统完全停止
  → 记录停机日志
  → 等待人工干预
```

## 余额检测方法

### 方法 1: API 错误监控

**检测指标**:
- 429 错误率（余额不足）
- 429 错误率（速率限制）
- 错误时间分布

**计算公式**:
```
余额风险等级 = (429 余额错误 / 总错误) × 100%

风险等级判定:
  0-10%: 🟢 正常
  10-30%: 🟡 关注
  30-50%: 🟠 警戒
  > 50%: 🔴 危急
```

### 方法 2: API 使用统计

**监控指标**:
- 每小时 API 调用次数
- Token 消耗速度
- 成本估算

**基线参考**:
```
正常状态（健康评分 9.5/10）:
  - API 调用: ~200 次/小时
  - Token 消耗: ~50,000 tokens/小时
  - 预计成本: ¥X/小时

警戒状态（健康评分 < 7.0）:
  - API 调用: < 50 次/小时
  - Token 消耗: < 10,000 tokens/小时
  - 大量 429 错误
```

### 方法 3: 日志分析

**检测模式**:
```bash
# 检查余额不足错误
grep "余额不足" /root/.openclaw/agents/main/sessions/*.jsonl | wc -l

# 检查速率限制错误
grep "速率限制" /root/.openclaw/agents/main/sessions/*.jsonl | wc -l

# 检查最近 1 小时错误
find /root/.openclaw/agents/main/sessions -name "*.jsonl" -mmin -60 \
  -exec grep -l "429\|余额不足" {} \; | wc -l
```

## 自动降级策略

### 任务优先级分类

| 优先级 | 任务 | 余额 > 20% | 余额 10-20% | 余额 < 10% |
|--------|------|-----------|------------|-----------|
| **P0 核心** | adaptive-scheduler | 正常 | 正常 | 最低频率 |
| **P0 核心** | evolver-self-evolution | 正常 | 降低 50% | 暂停 |
| **P1 重要** | evolver-log-analysis | 正常 | 降低 50% | 暂停 |
| **P1 重要** | evomap-auto-bounty | 正常 | 暂停 | 暂停 |
| **P2 一般** | autonomous-exploration | 正常 | 暂停 | 暂停 |
| **P2 一般** | novel-marketing-execute | 正常 | 暂停 | 暂停 |
| **P3 可选** | chaos-proxy | 正常 | 暂停 | 暂停 |

### 降级脚本模板

```bash
#!/bin/bash
# API 余额自动降级脚本

BALANCE_THRESHOLD_CRITICAL=10
BALANCE_THRESHOLD_WARNING=20

# 检测余额风险等级
check_balance_risk() {
  local balance_errors=$(grep "余额不足" /root/.openclaw/agents/main/sessions/*.jsonl 2>/dev/null | wc -l)
  local rate_limit_errors=$(grep "速率限制" /root/.openclaw/agents/main/sessions/*.jsonl 2>/dev/null | wc -l)
  local total_errors=$(($balance_errors + $rate_limit_errors))
  
  if [ $balance_errors -gt 50 ]; then
    echo "CRITICAL"
  elif [ $balance_errors -gt 20 ]; then
    echo "WARNING"
  else
    echo "NORMAL"
  fi
}

# 应用降级策略
apply_degradation() {
  local level=$1
  
  case $level in
    "CRITICAL")
      echo "[CRITICAL] 暂停所有非关键任务"
      # 暂停所有任务
      ;;
    "WARNING")
      echo "[WARNING] 降低任务频率 50%"
      # 降低频率
      ;;
    "NORMAL")
      echo "[NORMAL] 系统正常运行"
      # 正常运行
      ;;
  esac
}

# 主循环
risk_level=$(check_balance_risk)
apply_degradation $risk_level
```

## 通知机制

### 预警通知

**触发条件**:
- 余额 < 30%
- 429 余额错误 > 20 次/小时
- 连续 3 次任务失败

**通知内容**:
```markdown
## 🔔 API 余额预警

**时间**: 2026-02-26 04:00
**余额状态**: 低于 30%
**429 错误**: 25 次/小时
**系统健康评分**: 7.5/10

**建议行动**:
1. 检查 API 账户余额
2. 准备充值
3. 监控系统状态

**预计耗尽时间**: 约 6 小时
```

### 紧急通知

**触发条件**:
- 余额 < 10%
- 429 余额错误 > 50 次/小时
- 系统健康评分 < 7.0

**通知内容**:
```markdown
## 🚨 API 余额紧急警报

**时间**: 2026-02-26 04:30
**余额状态**: 低于 10%（危急）
**429 错误**: 136 次
**系统健康评分**: 6.5/10

**已采取行动**:
- ✅ 暂停所有非关键任务
- ✅ 降低核心任务频率
- ⏳ 等待充值

**紧急行动**:
1. **立即充值 API 账户**
2. 检查智谱 AI GLM-5 余额
3. 考虑切换到备用模型
```

## 成本优化建议

### 减少不必要的 API 调用

1. **优化任务频率**
   - 减少高频任务（如 evolver-log-analysis）
   - 合并相似任务
   - 使用缓存减少重复调用

2. **优化 Token 使用**
   - 控制上下文大小（< 50,000 tokens）
   - 清理过期会话
   - 使用更小的模型（如果可用）

3. **错峰执行**
   - 避开高峰期（07:00-09:00, 10:00, 15:00-16:00）
   - 在低峰期执行非关键任务

## 工具集成

### 与 peak-hours-monitoring 配合

- 在高峰期实施更严格的余额监控
- 余额 < 20% 时，高峰期暂停所有非关键任务
- 高峰期后恢复部分任务

### 与 api-retry-strategy 配合

- 余额不足时不重试（避免浪费 API 调用）
- 速率限制时使用指数退避
- 区分余额错误和速率限制错误

### 与 system-baseline-config 配合

- 将余额监控纳入基线配置
- 建立余额健康评分模型
- 定期更新成本基线

### 健康评分调整

```
原公式: 基础分 8.0 + 奖励分 - 惩罚分
新公式: 基础分 8.0 + 奖励分 - 惩罚分 - 余额惩罚

余额惩罚:
  - 余额 < 30%: -0.5
  - 余额 < 20%: -1.0
  - 余额 < 10%: -2.0
  - 余额 = 0: -3.0
```
