# SCL Control 层

## 职责

本 Skill 实现 SCL R-CCAM 中的 **Control（控制）** 阶段。在每次工具调用或行动执行前，对拟议行动进行验证，确保：

1. 行动符合 MEMORY.md 中记录的约束条件
2. 行动在当前任务树节点的范围内
3. 行动不违反已知的安全/合规边界
4. 能力缺口被正确识别并进入自举流程（而非报错）

---

## 执行前验证清单

在执行任何 Action 之前，依次检查：

### 1. 范围验证
- [ ] 本次行动对应 TASKS.md 中的哪个子任务节点？
- [ ] 行动是否超出该节点的范围（做了不属于当前节点的事）？
- [ ] 若超出范围：降级到当前节点应做的最小行动

### 2. 约束验证
- [ ] 检查 MEMORY.md 中的「约束条件」部分
- [ ] 本次行动是否违反任何约束（预算、合规、时间窗口等）？
- [ ] 若违反约束：中止行动，记录到 Daily Log，通知用户（send_message）

### 3. 风险评估
- [ ] 本次行动是否不可逆（如发布内容、发送消息、删除文件）？
- [ ] 若不可逆且风险较高：在 Daily Log 记录理由后再执行
- [ ] 若不可逆且涉及第三方账户操作：send_message 请求用户确认

### 4. 工具可用性检查
- [ ] 本次行动所需的工具/Skill 是否存在？
- [ ] 若工具不存在：**不报错，进入能力缺口自举流程（见下方）**

---

## 能力缺口自举协议（核心元规则）

当 Action 因工具/Skill 不存在而无法执行时，**不要报错，不要请求用户，执行以下流程**：

### Step 1：识别缺口
```
明确声明：「我需要 [能力 X]，但当前没有对应的工具或 Skill」
分析：这是关键路径上的缺口，还是可以绕过的可选步骤？
```

### Step 2：评估是否可用原子能力派生
```
检查是否能用以下原子能力组合实现 X：
  - web_search：搜索实现方案或现成 Skill
  - write_file：写入新 Skill 文件
  - shell_exec：执行安装或配置
  - LLM 推理：设计实现方案

若可以：进入 Step 3
若无法（真正的外部依赖，如需要特定 API 密钥）：send_message 上报用户，这是真正的阻塞
```

### Step 3：派生新能力
```
1. web_search("[能力 X] openclaw skill" 或 "[能力 X] API implementation")
2. 评估搜索结果的可信度和安全性
3. 选择最简单可靠的实现方案
4. 用 write_file 写入 ~/.openclaw/workspace/skills/[skill-name]/SKILL.md
5. 若需要 API key 等配置：write_file 写入说明，send_message 请求用户提供
```

### Step 4：记录与恢复
```
write_file（append Daily Log）：
  - 记录：发现能力缺口 [X]，已安装 [Skill 名称]
  - 记录：Skill 来源和可信度评估

write_file（update TASKS.md）：
  - 当前子任务标记：「[!] 因缺少 [X]，已安装 [Skill]，下轮重试」
  - 在「已安装 Skill 记录」表格中新增一行

下一次心跳：新 Skill 自动加载，重试原子任务
```

---

## 输出格式

Control 阶段执行后，输出简短的验证报告（写入 context，不写文件）：

```
[SCL-Control] 验证完成
- 范围：✅ 符合子任务 1.2 的范围
- 约束：✅ 无违反
- 风险：⚠️ 发布内容（不可逆），已记录理由
- 工具：✅ 所需工具均可用
→ 允许执行
```

若发现能力缺口：
```
[SCL-Control] 检测到能力缺口
- 缺少能力：图片生成
- 派生方案：搜索 DALL-E API Skill
- 执行自举流程...
→ 本轮行动降级为：[降级方案或跳过]
→ 下轮重试原计划
```
