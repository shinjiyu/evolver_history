# 目标设定

## 触发关键词

以下任意格式均触发本 Skill：

| 格式 | 示例 |
|------|------|
| `目标：[描述]` | `目标：帮我三个月内在小红书涨粉 1000` |
| `GOAL: [描述]` | `GOAL: grow my XHS account by 1000 followers` |
| `任务：[描述]` | `任务：持续监控竞品价格并每周汇报` |
| `/goal [描述]` | `/goal 帮我运营小红书账号` |
| 自然语言（长期意图）| `帮我运营小红书，目标是三个月涨粉 1000 人，不用付费推广` |

**注意**：普通的一次性问题（如「帮我查一下今天的天气」）不触发本 Skill。判断标准：
- 目标是否跨越多个心跳周期才能完成？→ 是，则触发
- 是否需要 Agent 持续跟踪进度？→ 是，则触发
- 是否是一次性请求（查询、计算、单步操作）？→ 否，不触发

## 执行流程

### Step 1：解析目标消息

从用户消息中提取以下要素：

```
顶层目标：[核心目标，一句话概括]
成功标准：[可量化的完成条件，如「涨粉 1000」「每周 3 篇」]
时间范围：[若用户提及，如「三个月内」；若未提及，标记「未指定」]
约束条件：[用户明确说明的限制，如「不用付费推广」「只做合规操作」]
领域上下文：[平台、账号、业务背景等]
```

若关键信息缺失（如完全没有成功标准或时间范围），**在 Step 5 的确认消息中明确提问**，等用户补充后再完整写入。

### Step 2：检查 TASKS.md 是否已有目标

```
读取 ~/.openclaw/workspace/TASKS.md：
  - 若「顶层目标」为空或是模板占位符 → 继续写入新目标
  - 若已有目标且与本次不同 → 在 Step 5 确认消息中提醒用户：
    「检测到已有目标：[旧目标]。
     是否替换为新目标？还是在此基础上新增子任务？」
    等待用户确认后再写入
  - 若已有目标且相同 → 告知用户「目标已存在，无需重复设定」
```

### Step 3：写入 TASKS.md

按以下格式写入（完全替换模板占位符，保留文件结构）：

```markdown
## 顶层目标
[顶层目标原文]

*目标设定时间*：YYYY-MM-DD HH:MM
*成功标准*：[可量化标准]
*时间范围*：[时间范围或「未指定」]

---

## 约束条件
- [约束 1]
- [约束 2]
- 若用户未提供约束：「暂无明确约束，执行中如遇边界情况将主动确认」
```

### Step 4：验证写入成功

**必须执行**：写入后立即用 read_file 读回 TASKS.md，验证以下内容：

```
✅ 「顶层目标」章节存在且内容非空
✅ 内容与 Step 1 解析的目标一致
✅ 「约束条件」章节存在
✅ 文件无损坏（可正常读取）
```

若验证失败（内容不一致或读取报错）：
- 重试写入一次
- 若仍失败：send_message 告知用户「TASKS.md 写入失败，请检查 workspace 文件权限」

### Step 5：向用户发送确认消息

通过 send_message 发送结构化确认：

```
✅ 目标已记录

📌 我理解的目标：
[顶层目标]

📏 成功标准：[成功标准]
⏱ 时间范围：[时间范围]
🚫 约束条件：
  · [约束 1]
  · [约束 2]

如有偏差，请直接回复「目标更正：[更正内容]」。

---
接下来我会：
1. 在下一次心跳中制定详细执行计划
2. 按心跳循环（每 [N] 分钟）自主推进，无需你持续指令
3. 遇到需要确认的情况会主动联系你
```

### Step 6：触发首次 ReCAP 分解

写入并确认后，立即调用 `recap-decompose` Skill，生成初始子任务树并写入 TASKS.md。

完成后追加到确认消息：

```
📋 初步执行计划：

[recap-decompose 生成的子任务树，简略版]

详细计划见 TASKS.md。
```

## 目标更正协议

若用户回复「目标更正：[内容]」，触发本 Skill 的更正流程：

1. 解析更正内容（是修改目标？还是补充约束？）
2. 更新 TASKS.md 对应字段
3. 验证写入
4. 重新触发 recap-decompose（若目标有实质性变化）
5. 发送更正确认消息

---

## 防止目标漂移的关键设计

TASKS.md 中的「顶层目标」是**只增不改**的原则的例外——它可以被 `goal-setting` Skill 修改，但**只有本 Skill 有权修改顶层目标**，其他 Skill（包括 scl-memory、recap-decompose）只能读取它。

这防止了 Agent 在执行过程中无意识地「重新定义目标」。

---

## 常见失败场景与处理

| 场景 | 处理 |
|------|------|
| Agent 把目标当普通对话 | SOUL.md 中的触发规则强制匹配关键词，本 Skill 介入 |
| TASKS.md 写入但内容错误 | Step 4 验证检测到，重试或通知用户 |
| 用户没说成功标准 | Step 5 确认消息中提问补充 |
| 已有旧目标 | Step 2 检测到，询问是替换还是叠加 |
| 目标太模糊无法分解 | recap-decompose 返回「需要更多信息」，在确认消息中列出需要澄清的问题 |
