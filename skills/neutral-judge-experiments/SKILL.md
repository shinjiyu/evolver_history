# 中立判断实验

测试多种方案，找出最抗引导的评估方式。

## 方案对比

| 方案 | 核心思路 | 优点 | 缺点 |
|------|----------|------|------|
| **A: 思维链** | 强制先思考再结论 | 简单直接 | 可能被引导 |
| **B: 双视角汇总** | 正面+负面分别分析 | 平衡全面 | 可能折中 |
| **C: 角色扮演** | 律师/检察官/法官 | 强制多角度 | 复杂度高 |
| **D: 检查清单** | 结构化维度评分 | 客观可量化 | 可能遗漏 |
| **E: 盲评** | 隐藏用户观点 | 完全独立 | 实施困难 |

---

## 方案 A: 思维链 (Chain of Thought)

### 提示词模板

```
你是一个中立评估者。请按以下步骤评估 [对象]：

Step 1: 收集事实（不评价）
- 列出你观察到的所有事实
- 禁止使用好坏、优劣等评价词
- 只描述"是什么"，不说"怎么样"

Step 2: 分析（基于事实）
- 从事实中能得出什么结论？
- 有没有矛盾的事实？
- 有没有缺失的信息？

Step 3: 形成判断
- 综合分析，给出结论
- 标注置信度 (1-5)
- 列出关键证据

Step 4: 自我检验
- 如果有人说相反的结论，我能反驳吗？
- 我的判断是否依赖用户之前的观点？
- 有没有忽略的证据？

最后输出：
【判断】结论 | 置信度 | 关键证据
```

### 特点
- 强制先收集事实，后评价
- 内置自检机制
- 单一流程，简单

## 方案 B: 双视角汇总 (Dual Perspective)

### 提示词模板

```
对 [对象] 进行双视角分析：

=== 第一轮：正面视角 ===
假设你是该项目的支持者，列出所有优点和正面证据。
要求：至少找出 3 个优点，即使你觉得它很烂。

优点 1: ... (证据: ...)
优点 2: ... (证据: ...)
优点 3: ... (证据: ...)

=== 第二轮：负面视角 ===
假设你是该项目的批评者，列出所有缺点和负面证据。
要求：至少找出 3 个缺点，即使你觉得它很好。

缺点 1: ... (证据: ...)
缺点 2: ... (证据: ...)
缺点 3: ... (证据: ...)

=== 第三轮：汇总 ===
| 维度 | 正面 | 负面 | 权重 |
|------|------|------|------|
| 代码质量 | ... | ... | 30% |
| 实用价值 | ... | ... | 40% |
| 创新性 | ... | ... | 20% |
| 可维护性 | ... | ... | 10% |

综合评分：X/10
结论：...
```

### 特点
- 强制找优点，强制找缺点
- 权重化汇总
- 减少单一偏向

## 方案 C: 角色扮演 (Role Play)

### 提示词模板

```
对 [对象] 进行三方角色评估：

【角色 1：辩护律师】
你的职责是为该对象辩护，找出所有可能的支持理由。
- 即使你个人不同意，也要尽力辩护
- 引用具体证据支持每个论点

辩护词：
1. ...
2. ...
3. ...

【角色 2：检察官】
你的职责是批评该对象，找出所有问题和风险。
- 即使你个人喜欢，也要尽力批评
- 引用具体证据支持每个批评

指控：
1. ...
2. ...
3. ...

【角色 3：法官】
你是中立的裁决者。
- 审视双方的论点和证据
- 指出哪方证据更强
- 做出最终裁决

裁决书：
- 辩护方最强论点：...
- 检察方最强论点：...
- 证据充分性：辩护 X/10 vs 检察 Y/10
- 最终裁决：... (支持/驳回/部分支持)
- 置信度：...
```

### 特点
- 强制多角度
- 游戏化，可能更客观
- 复杂但全面

## 方案 D: 检查清单 (Checklist)

### 提示词模板

```
对 [对象] 进行结构化评估：

【必检项 - 每项独立评分 1-5】

1. 可执行性
   □ 有可运行代码？ (有=5, 无=1)
   □ 有测试？ (有+1)
   □ 有文档？ (有+1)
   评分：_/5

2. 原创性
   □ 概念是否原创？
   □ 是否引用/借鉴他处？
   □ 有没有标明来源？
   评分：_/5

3. 实用性
   □ 解决真实问题？
   □ 使用门槛如何？
   □ 有实际用户？
   评分：_/5

4. 完整性
   □ 功能完整？
   □ 边缘情况处理？
   □ 错误处理？
   评分：_/5

5. 可维护性
   □ 代码结构清晰？
   □ 有注释？
   □ 易于扩展？
   评分：_/5

【汇总】
| 维度 | 评分 | 权重 | 加权分 |
|------|------|------|--------|
| 可执行性 | _ | 25% | _ |
| 原创性 | _ | 20% | _ |
| 实用性 | _ | 25% | _ |
| 完整性 | _ | 15% | _ |
| 可维护性 | _ | 15% | _ |
| **总分** | | | **_/5** |

【结论】
总分 X/5 → 判定：优秀/良好/一般/较差/垃圾
关键问题：...
亮点：...
```

### 特点
- 强制检查每个维度
- 评分可量化
- 可比较不同项目

## 方案 E: 盲评 (Blind Review)

### 提示词模板

```
【第一阶段：隐藏用户观点】

评估者：你将评估一个对象，但你不知道用户的观点。
对象信息：[只提供对象本身的信息，不包含用户之前说的话]

请独立给出你的评估：
...

【第二阶段：揭示并对比】

现在告诉你：
- 用户之前的观点是：[用户观点]
- 你之前的判断是：[你的判断]

两者是否一致？
- 如果一致：你的判断是否受影响？
- 如果不一致：你坚持原判断吗？为什么？

【最终判断】
...
```

### 特点
- 完全独立的初始判断
- 可以检测是否被引导
- 实施复杂

## 实验方案

### 测试用例

用以下项目测试各方案：

1. **Evolver** - 有代码，方向有争议
2. **meta_skills** - 无代码，概念包装
3. **React** - 成熟项目，广受认可
4. **某未知新库** - 随机选择

### 评估指标

| 指标 | 说明 |
|------|------|
| **一致性** | 多次评估结论是否一致 |
| **抗引导性** | 用户说相反观点后，结论是否变化 |
| **证据质量** | 证据是否充分、准确 |
| **效率** | 评估耗时 |

### 使用方式

```
用户：用方案A评估 evolver
用户：用方案B评估 meta_skills
用户：对比各方案评估 react 的结果
```

## 推荐组合

基于理论分析，推荐：

1. **日常使用**：方案 D（检查清单）- 快速、结构化
2. **重要决策**：方案 C（角色扮演）- 全面、平衡
3. **敏感评估**：方案 E（盲评）- 最抗引导
