## 🎯 项目核心

### 游戏概念
- **3 个 AI Agent** 组成探险队
- **探索随机地牢**（7 个房间，多种类型）
- **协作与冲突**：Agent 有个人目标，需要协作也需要博弈
- **成长系统**：经验、升级、装备、记忆

### 人类角色
- **观察者**：观看 AI "玩游戏"
- **顾问**：提供建议（Agent 会考虑但不一定听从）
- **指挥官**：关键时刻接管某个 Agent

---

## ✨ 核心特性

### 1. 智能 Agent

每个 Agent 有：
- **性格**：勇敢、谨慎、贪婪、利他
- **目标**：个人和团队目标
- **记忆**：短期记忆（当前探险）+ 长期记忆（历史）
- **关系**：与其他 Agent 的信任度

**示例**：
```javascript
{
  name: "凯恩",
  role: "warrior",
  personality: {
    bravery: 0.9,    // 勇敢：倾向冒险
    caution: 0.3,    // 谨慎：较少考虑风险
    greed: 0.6,      // 贪婪：喜欢宝物
    altruism: 0.5    // 利他：平衡自我和团队
  },
  goals: ["保护队友", "击败敌人", "探索地牢"]
}
```

### 2. 随机地牢

**程序化生成**：
- 入口 → 大厅 → 战斗 → 宝藏 → 陷阱 → 休息 → 神秘 → Boss
- 随机敌人（哥布林、骷髅兵、史莱姆、暗影狼...）
- 随机物品（药水、武器、魔法卷轴...）
- 随机事件（谜题、祝福、诅咒...）

### 3. 战斗系统

**回合制战斗**：
- Agent 选择攻击目标
- 计算伤害（攻击 - 防御）
- 敌人反击
- 击败敌人获得经验和战利品

### 4. 成长系统

- **经验值**：击败敌人获得经验
- **升级**：属性提升，学习新技能
- **装备**：拾取物品增强实力
- **记忆**：从历史中学习

## 🏗️ 技术架构

### 核心组件

```
Game Orchestrator (编排器)
    ├── 管理 3 个 Agent Tasks
    ├── 协调游戏循环
    ├── 保存/加载游戏状态
    └── 生成探险报告

Agent Task (智能体)
    ├── 感知：读取游戏状态
    ├── 决策：使用 AI 做决策
    ├── 执行：执行动作
    └── 记忆：记录重要事件

Dungeon Generator (地牢生成器)
    ├── 程序化生成房间
    ├── 放置敌人和物品
    └── 连接房间
```

### 数据流

```
1. Orchestrator 初始化地牢
2. 创建 3 个 Agent tasks
3. 每回合：
   a. 更新感知
   b. Agent 并行决策
   c. 按速度执行动作
   d. 更新游戏状态
   e. 记录事件
4. 检查游戏结束条件
5. 生成探险报告
```

## 🎓 设计理念

### 为什么这个设计？

1. **展示 Agent 智能**
   - 协作、冲突、决策、成长都能体现
   - 不是简单脚本，而是有目标、性格、记忆的"智能体"

2. **技术可行性**
   - 完美适配 OpenClaw task 系统
   - 每个 Agent 可以是独立 task
   - 状态简单，易于调试

3. **可玩性**
   - 随机地牢：每次都不同
   - 策略深度：不同 Agent 组合产生不同策略
   - 叙事性：Agent 的决策和互动产生独特故事

4. **人类参与**
   - 低门槛：观察即可
   - 高影响：建议能改变 Agent 决策
   - 有趣：观看"AI 玩游戏"本身就是娱乐

### 核心创新

1. **Agent 驱动**：不是人类玩游戏，是 AI 玩游戏
2. **观察者模式**：人类是观众，偶尔提建议
3. **协作 + 冲突**：Agent 既合作又竞争
4. **成长记忆**：Agent 有长期记忆和成长

## 💡 关键设计决策

### 为什么选择协作探险？
- 展示协作和冲突
- 每个 Agent 有明确角色
- 容易理解和欣赏

### 为什么使用文件系统？
- 简单，无需数据库
- 可观察，直接查看 JSON
- 持久化，断电不丢失

### 为什么 MVP 用简单 AI？
- 快速验证核心机制
- 不依赖外部 API
- 为 LLM 决策提供对比
