## 🎯 设计亮点

### 1. 智能化 Agent
每个 Agent 不是简单的脚本，而是有：
- **性格驱动决策**：不同性格导致不同行为
- **目标导向**：Agent 追求个人和团队目标
- **记忆积累**：从历史事件中学习

### 2. 随机性和可重玩性
- 每次地牢都不同
- 敌人类型和数量随机
- 物品和事件随机
- Agent 决策有随机因素

### 3. 协作与冲突
- Agent 需要协作才能通关
- 但 Agent 有个人目标，可能产生冲突
- 关系系统影响信任和合作

### 4. 技术可扩展性
- 模块化设计
- 易于添加新的房间类型、敌人、物品
- 可以集成真正的 LLM 决策
- 支持 Web UI 扩展

## 💡 关键设计决策

### 为什么选择这个方向？
1. **展示 Agent 智能**: 协作、冲突、决策、成长都能体现
2. **技术可行性**: 完美适配 OpenClaw task 系统
3. **可玩性**: 随机地牢 + 策略深度 + 叙事性
4. **人类参与**: 低门槛（观察），高影响（建议）

### 为什么使用文件系统作为状态存储？
1. **简单**: 无需数据库，易于调试
2. **可观察**: 直接查看 JSON 文件
3. **持久化**: 自动保存，断电不丢失
4. **扩展性**: 后续可迁移到数据库

### 为什么 MVP 使用简单决策逻辑？
1. **快速验证**: 不依赖外部 API，可立即测试
2. **可控**: 行为可预测，便于调试
3. **基准**: 为后续 LLM 决策提供对比
4. **演示**: 可以展示游戏核心机制

## 🎓 学到的经验

### 1. 模块化设计的价值
每个组件（Orchestrator、Agent、DungeonGenerator）独立，易于：
- 单独测试
- 替换实现
- 扩展功能

### 2. 日志和可观察性的重要性
详细的事件日志帮助：
- 调试问题
- 理解 Agent 行为
- 生成报告

### 3. MVP 思维
先实现核心功能，再逐步增强：
- 核心循环 → 战斗系统 → LLM 集成
- 简单决策 → 规则 AI → LLM 决策

### 4. 游戏设计的平衡
- 简单 vs 复杂
- 随机 vs 可控
- 协作 vs 冲突
- 人类 vs AI
